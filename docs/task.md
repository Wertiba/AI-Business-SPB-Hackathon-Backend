# NeuroICE — AI Business SPB Hackathon 2026

Задача — детекция аномалий в записях холостого хода двигателя на уровне 5-секундных чанков. На вход — предварительно нарезанные WAV-чанки (5 сек, 48 kHz, stereo, 24-bit). Нужно вернуть скор аномальности (float, больше = более аномально) для каждого чанка.

Тренировочные данные содержат только нормальные записи. Тестовые данные содержат нормальные и аномальные записи. Тестовая разметка командам недоступна — вы отправляете код, который запускается в серверной песочнице.

**Платформа:** https://app.ai-business-spb.ru

---

## Описание задачи

ООО «НейроДВС» разрабатывает AI-систему акустической диагностики технического состояния двигателей. Подход основан на анализе аудиосигналов методами машинного обучения.

**Проблема.** В транспортных компаниях и автопарках большинство неисправностей двигателя диагностируется только после появления явных симптомов или при плановом обслуживании. Это приводит к неожиданным поломкам, простою техники, дорогостоящему ремонту и рискам аварий.

**Роль ИИ.** Каждый двигатель имеет характерный акустический «портрет». При возникновении неисправностей изменяются частотный спектр звука, амплитуда колебаний и временные паттерны сигнала. Методы машинного обучения позволяют автоматически обнаруживать такие изменения — с точностью и скоростью, недостижимыми для традиционных методов диагностики.

## Цель и результат хакатона

**Цель:** разработать ML-алгоритм для анализа акустических сигналов двигателя и выявления отклонений от нормального режима работы.

**Ожидаемый результат:** работающая модель, которая принимает на вход WAV-файл записи двигателя и возвращает числовой `anomaly_score` — чем выше значение, тем выше вероятность аномалии. Модель обучается только на нормальных записях (unsupervised / one-class подход).

## Потенциал бизнес-внедрения

Раннее выявление отклонений в работе двигателя позволяет:

* снизить затраты на ремонт за счёт перехода от реактивного к предиктивному обслуживанию;
* предотвращать поломки и незапланированные простои техники;
* увеличивать ресурс двигателей и продлевать срок службы парка;
* повышать безопасность эксплуатации транспортных средств.

## Требования к решению

**Тип задачи:** anomaly detection без разметки аномалий (unsupervised / one-class).

**Вход:** WAV-файл записи работы двигателя (5-20 сек, дискретизация 48 kHz). Обучение — только на записях «норма».

**Выход:** числовой `anomaly_score` (float) — больше значение, более аномальна запись.

**Что анализирует модель:**

* аудиозаписи работы двигателя;
* спектральные характеристики сигнала;
* статистические признаки временного ряда.

**Что определяет модель:**

* нормальный режим работы двигателя;
* подозрение на неисправность;
* аномальное поведение.

**Область задачи:** machine learning, audio signal processing, predictive maintenance.

**Технические ограничения по производительности:**

| Сценарий | Лимит |
|----------|-------|
| Инференс одной записи (CPU) | ≤ 10 секунд |
| Пакетная обработка 1 000 записей | ≤ 3 часа |
| Sandbox: CPU / RAM / время | 6 CPU / 16 GB / 600 сек |

## Особенности задачи — важно учесть

**Целевой двигатель:** дизельный, автобусный парк (Weichai); условия записи — зона ТО с фоновым шумом.

**Ключевое ограничение:** размеченных аномалий нет → unsupervised / self-supervised (one-class) подход является предпочтительным.

**Главный технический риск — RPM как конфаундер.** Изменения спектральных признаков могут быть вызваны разными оборотами двигателя, а не дефектом. Поле `rpm_mode` есть только в трейн-манифесте; на инференсе фильтровать по нему нельзя. Участникам рекомендуется учитывать этот фактор при построении признаков и модели.

## Дополнительные баллы

Жюри может присудить бонусные баллы за:

**Устойчивость к шуму** — модель сохраняет качество при наличии фонового шума в записях (характерно для условий зоны ТО).

**Простая объяснимость** — указание, какие частоты или временные участки повлияли на вердикт: аннотированный график спектра или аналогичная визуализация. Полноценный XAI не требуется.

**Переносимость на другие типы ДВС (domain holdout)** — качество модели на записях двигателей, не входивших в обучающую выборку.

**Приоритет жюри при равных метриках:** интерпретируемость решения ценится выше сложности архитектуры.

## Метрика

```
roc_auc          = ROC-AUC по chunk-level anomaly_score
recall_at_fpr5   = recall при FPR = 5%
combined_score   = 0.60 * roc_auc + 0.40 * recall_at_fpr5
```

Дополнительно: `recall_at_fpr1`, `pr_auc` (информационные). `combined_score` — ключ лидерборда.
